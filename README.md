## 项目背景

这个项目源于一个我曾深度参与的开源微服务社区项目 Sea-RideTheWind。在该项目中，我负责 **Article 服务** 的整体架构设计与核心功能实现——从数据建模、API 设计到消息队列集成、缓存策略落地。

2026 年 5月，原组织解散，项目停止维护。我将自己负责的文章服务代码完整剥离，独立 fork 至本仓库，进行了以下重构工作：

- **剥离微服务依赖**：去除对用户、评论、点赞等外部服务的 RPC 调用，文章服务完全自治
- **重新设计数据边界**：所有文章相关数据（正文、标签、计数、状态）收敛至独立 PostgreSQL 实例
- **优化消息链路**：引入 Transactional Outbox 模式替代原有的直接 Kafka 写入，解决 DB 与消息队列的一致性问题
- **自研分布式限流**：替换 go-zero 原生限流器，解决其在 Redis 哨兵集群下的兼容性问题
- **完善可观测体系**：接入 Prometheus + OpenTelemetry + Grafana，实现全链路指标与追踪

本项目由我独立维护，持续迭代至今。

## 核心架构与难点攻克

### 1. Transactional Outbox —— DB 与 Kafka 的最终一致性

**问题**：文章创建需要同时写入 PostgreSQL 和发送 Kafka 消息。直接在同一事务中操作两者，存在 DB 成功但 Kafka 投递失败的风险，导致数据不一致。

**方案**：采用 **Transactional Outbox 模式**。

- 在 DB 事务中，文章数据与 Outbox 事件写入同一张表，保证原子性
- 独立的 **Relay 进程** 轮询 Outbox 表，将未投递事件发送至 Kafka
- 投递成功后标记事件为已发送，支持重试与死信处理

**效果**：DB 与 Kafka 消息达到最终一致性，不依赖分布式事务，故障恢复能力显著提升。

### 2. Kafka 消费幂等 —— 三重保障

**问题**：Kafka 消费者可能因重试、Rebalance 等原因重复消费同一条消息，导致数据重复写入。

**方案**：三层幂等防线。

| 层级 | 机制 | 作用 |
|:--|:--|:--|
| 第一层 | **Redis 分布式锁** | 同一事件 Key 并发消费互斥 |
| 第二层 | **状态机校验** | 检查当前状态是否允许本次操作，非法状态直接拒绝 |
| 第三层 | **事件 Key 唯一索引** | DB 层兜底，重复事件 Key 写入直接冲突 |

**效果**：任何层级的重复消息都不会产生脏数据，消费端具备端到端幂等保证。

### 3. 自研分布式令牌桶限流器

**问题**：go-zero 内置的 `TokenBucketLimit` 在 Redis 哨兵集群（1 主 2 从 3 哨兵）模式下存在连接兼容问题，无法正常工作。

**方案**：基于 **Redis Lua 脚本** 自研分布式令牌桶。

- 令牌补充与扣减在 Lua 脚本中原子执行，避免并发竞争
- 支持配置桶容量、令牌补充速率、单次请求消耗令牌数
- 天然兼容 Redis 哨兵与集群模式

**效果**：限流精度达到毫秒级，支持多节点共享同一限流状态，生产环境零超卖。

### 4. 多级缓存架构 —— 击穿 / 雪崩 / 穿透全防

**问题**：单一 Redis 缓存层在高并发场景下存在热点 Key 击穿、大面积过期雪崩、空查询穿透三重风险。

**方案**：**进程内 TTL 缓存 + Redis** 两级缓存，配合三重防护策略。

```
请求 → 本地缓存 (命中?) → Redis (命中?) → DB
                ↓ miss         ↓ miss
           Singleflight     空值缓存
           合并并发请求       防穿透
```

| 策略 | 机制 | 解决问题 |
|:--|:--|:--|
| **Singleflight** | 同一 Key 的并发请求合并为一次 DB 查询 | 缓存击穿 |
| **随机 TTL** | 过期时间加入随机偏移量，打散集中失效 | 缓存雪崩 |
| **空值缓存** | 查询结果为空时写入短 TTL 占位值 | 缓存穿透 |
| **Pub/Sub 失效** | 通过 Redis Pub/Sub 广播缓存失效事件，各节点清除本地缓存 | 跨节点一致性 |

**效果**：本地缓存命中率 > 85%，Redis 命中率 > 95%，DB 查询压力降低 90%+。

### 5. 高性能异步计数方案

**问题**：文章浏览量如果每次请求都直接写 DB，在高并发下会产生严重的写放大和锁竞争。

**方案**：**Redis 脏集合 + 计数 Key + Lua 原子弹出** 的异步增量计数架构。

- 用户访问文章时，仅将文章 ID 写入 Redis **脏集合**（Dirty Set），浏览量计数 Key 原子递增
- 定时任务从脏集合中批量弹出文章 ID（Lua 脚本保证原子性），读取计数 Key 的增量值，回刷 PostgreSQL
- 回刷完成后清除脏标记，支持断点续跑

**效果**：DB 写入频率从每请求一次降至定时批量回刷（可配置周期），单机浏览量写入 QPS 从 DB 层完全卸载至 Redis。

### 6. 全链路可观测

**问题**：微服务架构下，一次请求跨多个组件，故障定位困难。

**方案**：三位一体的可观测体系。

- **Prometheus**：采集业务指标（文章创建/查询 QPS、Kafka 消费延迟、缓存命中率等），自定义 Grafana 大盘
- **OpenTelemetry**：链路追踪覆盖 HTTP 入口 → 业务逻辑 → DB → Redis → Kafka 全链路
- **Grafana**：统一仪表盘，支持按服务、接口、错误码多维度下钻

## 快速开始

### 环境准备

```bash
# 基础依赖
Go 1.25+
Docker / Docker Compose
goctl (go-zero 代码生成工具)
```

### 启动基础设施

```bash
docker compose up -d
```

启动 PostgreSQL、Redis 哨兵集群、Kafka、MinIO、etcd、Prometheus、Grafana、Jaeger。

### 启动文章服务

```bash
# 生成代码（如需）
goctl api go -api api/article.api -dir . -style go_zero

# 启动服务
go run service/article/main.go -f service/article/etc/article.yaml
```

### 验证服务

```bash
curl http://localhost:8080/article/ping
```
